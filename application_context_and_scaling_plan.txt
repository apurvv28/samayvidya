TIMETABLE APPLICATION: COMPLETE CONTEXT + SCALING PLAN
Date: April 10, 2026

============================================================
1) CURRENT PROTOTYPE CONTEXT (WHAT EXISTS TODAY)
============================================================

A. Product Purpose
- This project is a timetable scheduling platform for academic institutions.
- It supports coordinator workflows to configure master data, ingest teaching load, generate timetables, review versions, edit entries, and export PDFs.
- It includes role-oriented dashboards (coordinator, faculty, student) in the frontend.

B. Workspace Layout
- Root data and outputs:
  - Multiple CSVs (for divisions/years) used as source inputs and snapshots.
  - output.json for generated/result data.
  - existing tt/ and generated tt/ folders for timetable artifacts.
- Backend:
  - FastAPI application with modular routers, services, schemas, and Supabase integration.
  - Dockerfile and docker-compose.yml for containerized local/dev deployment.
- UI:
  - Next.js app-router frontend with Tailwind CSS and Supabase client SDK.

C. Backend Stack and Runtime
- Core stack:
  - FastAPI + Uvicorn
  - Supabase Python client
  - Pydantic Settings (.env based config)
  - JWT/Auth dependencies
  - ReportLab + Pillow for PDF generation
  - LiteLLM + AWS Bedrock hooks (optional pass-level planning signals)
  - CrewAI/CrewAI tools for load-management agent workflows
- Main API entry:
  - backend/app/main.py initializes FastAPI, CORS, startup/shutdown logs, health endpoint, and all routers.
- Config model:
  - backend/app/config.py expects Supabase credentials, SMTP credentials, app mode flags, and Bedrock model/region.

D. API Capabilities (Router-Level)
- /auth: authentication endpoints.
- /departments, /divisions, /subjects, /faculty, /rooms, /batches: master-data management.
- /days, /time-slots: scheduling calendar/time configuration.
- /faculty-leaves, /campus-events, /academic-years: operational constraints and metadata.
- /timetable-versions: versioned timetable metadata and listing.
- /timetable-entries: timetable row CRUD (with conflict protection against faculty/room double-booking).
- /agents:
  - input readiness diagnostics (preflight integrity checks)
  - default seed setup (days, slots, preferred faculty windows)
  - faculty load summarization
  - timetable generation (sync and SSE stream modes)
- /pdf: branded timetable export with institutional styling.

E. Scheduling Engine Design (backend/app/services/timetable_orchestrator.py)
- Core object:
  - TimetableOrchestrationEngine
- Functional pattern:
  - Multi-pass, deterministic scheduling with optional LLM hooks for pass-level traces/signals.
- Key preprocessing behaviors:
  - Normalization utilities for faculty, division, subject matching.
  - Converts load rows into session tasks for THEORY, LAB, and TUTORIAL.
  - Theory can be grouped into 2-slot contiguous blocks plus remainder 1-slot sessions.
  - Labs are 2-slot contiguous blocks and can be batch-specific.
- Constraint system (current behavior from code + repo memory):
  - Working days trimmed to weekday schedule (typically Mon-Fri; Sat/Sun non-working defaults seeded).
  - Sunday non-working validation gate.
  - Gapless day pattern enforcement.
  - Lunch break handling integrated in day pattern logic.
  - Division full-session vs batch-session overlap guards.
  - Faculty single-slot conflict guard.
  - Room occupancy conflict guard.
  - Session/day cap and daily theory+lab hard limits.
  - Strict parallel lab scheduling for feasible multi-batch groups (with feasibility guard).
  - Preference-based room selection by type and usage balancing.
  - Deterministic ordering/mixed scheduling strategy for better coverage.
- Outputs:
  - Scheduled entries + unresolved tasks diagnostics.
  - Pass traces, stage-wise metrics, and optional persisted timetable version/entries.

F. Data Model and Persistence Approach
- Supabase acts as primary operational database and auth/session ecosystem.
- Master/reference entities include:
  - departments, divisions, subjects, faculty, rooms, batches, days, time_slots, academic_years.
- Planning/execution entities include:
  - load_distribution, timetable_versions, timetable_entries, faculty_leaves, campus_events.
- There is SQL scaffolding in sql/ (example: create_subjects_table.sql).

G. Frontend Context (ui/)
- Next.js 16 + React 19 + Tailwind 4 stack.
- Key UX areas:
  - Landing pages and auth pages.
  - Coordinator dashboard tabs:
    - semester setup
    - faculty/load management
    - division management
    - resources (rooms/batches etc.)
    - agent orchestrator trigger
    - timetable viewer for latest version navigation
  - Faculty and student dashboards.
- API integration:
  - Uses NEXT_PUBLIC_API_BASE_URL (defaults to localhost backend).
  - Coordinator view retrieves latest timetable version and refreshes when needed.

H. PDF and Reporting Context
- PDF export is not basic dump output; it is branded and polished.
- Division-first rendering reduces overlap and improves readability.
- Includes metadata (version, semester, validity range), styled grid, and generated footer details.

I. Dev/Test/Local Ops Context
- Backend includes a test runner script (backend/test_timetable.py) for orchestration dry runs.
- Docker assets exist for backend containerization.
- UI uses webpack mode for dev stability on Windows (to avoid Turbopack instability in this project).

J. Current Prototype Strengths
- End-to-end loop exists already: data setup -> automated generation -> versioned result -> manual correction -> export.
- Good operational constraints are already encoded in scheduler logic.
- Architecture is modular enough to scale by isolating scheduler, APIs, and UI.

K. Current Prototype Gaps Before Full-Scale Institutional Rollout
- Single-database/project assumptions may need hard multi-tenant separation.
- Limited explicit enterprise observability, audit, SSO/RBAC depth, and compliance controls.
- Need stronger asynchronous job orchestration for large institutions and concurrent generations.
- Need hardened CI/CD, environment promotion, and disaster recovery strategy.
- Need formalized data contracts, migration strategy, SLAs, and support workflows.


============================================================
2) HOW TO SCALE THIS PROTOTYPE INTO A FULLY DEPLOYED,
   END-TO-END TOOL FOR SCHOOLS AND COLLEGES
============================================================

A. Goal for the Next 1 Month
- Deliver a usable end-to-end deployment for one institution (or pilot department) on low/no cost infrastructure.
- Keep architecture simple, stable, and free-tier friendly.
- Support: login, master-data setup, timetable generation, manual edits, PDF export, and publish/share.


B. Free-Tier Target Architecture (AWS + Existing Supabase)

1) Keep Supabase as the managed DB/Auth layer (already integrated)
- Reason: avoids managing RDS/Auth in month 1 and keeps costs low.

2) Backend hosting (FastAPI)
- Primary option (AWS Free Tier): EC2 t2.micro/t3.micro (Amazon Linux/Ubuntu) + Docker.
- Run backend container with environment variables and reverse proxy via Nginx.

3) Frontend hosting (Next.js)
- Free-tier approach A (simplest): Host static/export-compatible frontend on S3 static website + CloudFront.
- Free-tier approach B (if SSR is required): Deploy frontend on same EC2 instance using Node.js + PM2.
- Recommended for month 1: same EC2 instance for UI + API to reduce ops complexity.

4) File storage and backups
- Use S3 Free Tier for generated PDF artifacts and backup exports.

5) DNS and TLS
- Route 53 hosted zone (not strictly free) can be skipped in month 1 by using EC2 public DNS.
- If custom domain is required, use low-cost domain + Cloudflare free DNS/TLS proxy.


C. LLM via Amazon Bedrock (Month-1 Practical Setup)
- Your backend already supports Bedrock model settings via config (region/model).
- Use Bedrock only for optional pass-level advisory hooks; deterministic scheduler remains source of truth.
- Important cost note:
  - Bedrock usage is pay-per-token and may not be fully covered by AWS Free Tier.
  - For near-free operation in month 1, enforce strict usage controls:
    - Keep max_tokens small (already low in code).
    - Enable LLM only for selected runs (feature flag in UI/API).
    - Add daily budget alarms and request caps.
    - Fall back automatically to deterministic mode if cap is hit.


D. CI/CD on Free Tier: GitHub Actions or Jenkins

Option 1: GitHub Actions (recommended)
- Why: zero infra maintenance and free minutes for small private/public usage (subject to plan limits).
- Pipeline stages:
  - backend lint/test
  - frontend lint/build
  - Docker image build
  - deploy to EC2 via SSH (pull latest code, rebuild containers, restart)
- Secrets to configure in GitHub:
  - SUPABASE_URL, SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY
  - SMTP_* vars
  - BEDROCK_REGION, BEDROCK_MODEL
  - EC2_HOST, EC2_USER, EC2_SSH_KEY

Option 2: Jenkins on EC2 (only if needed)
- Why: full control but more maintenance overhead.
- Deploy Jenkins on same free-tier EC2 or a second small instance (higher risk of resource exhaustion).
- Use webhook-triggered pipeline for test/build/deploy.
- Month-1 recommendation: prefer GitHub Actions unless strict local CI requirement exists.


E. 4-Week Execution Plan (End-to-End in 1 Month)

Week 1: Production Baseline
- Prepare production .env templates for backend and frontend.
- Add health checks for UI/API and validate all critical routes.
- Finalize CORS and origin settings for deployed domain/IP.
- Add seed script runbook for days/time slots/departments/divisions/faculty/rooms.

Week 2: AWS Deployment
- Provision EC2 free-tier instance and harden it:
  - non-root user, firewall/security groups, fail2ban (optional), key-only SSH.
- Deploy backend container and frontend runtime.
- Configure Nginx reverse proxy:
  - /api -> FastAPI
  - / -> frontend
- Verify Supabase connectivity and PDF generation in deployed environment.

Week 3: CI/CD and Bedrock Controls
- Implement GitHub Actions workflow:
  - on push to main: test -> build -> deploy.
- Add rollback script:
  - keep previous container tag and quick restore command.
- Enable Bedrock with guardrails:
  - config toggle to disable/enable hooks
  - token cap and budget alert via CloudWatch Billing alarms.

Week 4: Stabilization and Pilot Go-Live
- Run pilot dataset end-to-end with real users.
- Capture and fix top issues:
  - unresolved session diagnostics
  - room/faculty conflict edge cases
  - UI usability in coordinator flow.
- Prepare handover docs:
  - admin setup guide
  - operator runbook
  - incident/recovery checklist.


F. Free-Tier Cost Control Checklist (Critical)
- EC2: stay on single t2.micro/t3.micro; stop nonessential services.
- S3: lifecycle policy for old PDFs/logs.
- CloudWatch: minimal log retention (e.g., 7-14 days for month 1).
- Bedrock: hard request cap + small token limit + disable by default for normal runs.
- CI: avoid excessive workflow runs; trigger deploy only on main branch.


G. Minimum Production Readiness for Month-1 Launch
- Must-have functional checks:
  - login/auth works
  - master data CRUD works
  - agent readiness + timetable generation works
  - manual timetable edits save with conflict protection
  - PDF export works.
- Must-have reliability checks:
  - service restarts recover automatically (systemd/compose)
  - daily DB export or backup snapshot exists
  - error logs are accessible
  - one-step rollback process is documented.


H. Recommended Month-1 Decisions
- Keep single-tenant pilot scope first (one school/college).
- Keep Supabase managed DB/Auth instead of migrating to RDS/Cognito now.
- Use GitHub Actions first; move to Jenkins only if mandated.
- Use Bedrock sparingly as optional intelligence layer, not required for core scheduling.


============================================================
3) FINAL SUMMARY
============================================================

For the next 1 month, this project can be made truly end-to-end with a pragmatic free-tier-first setup: FastAPI + Next.js deployed on one AWS free-tier EC2 instance, Supabase retained for DB/Auth, S3 for artifacts, and CI/CD via GitHub Actions (or Jenkins if required). Bedrock can be integrated for optional LLM assistance, but should be tightly usage-capped because Bedrock is generally metered.

This plan prioritizes fast deployment, low cost, operational simplicity, and pilot readiness over heavy enterprise redesign, which is the correct strategy for a one-month go-live target.
